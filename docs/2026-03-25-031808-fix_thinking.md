     Fix: Interactive TUI stuck on "Thinking..."

     Context

     The interactive TUI (nanobot agent without -m) gets stuck showing
     "Thinking..." and never displays the LLM response. There are two root
     causes plus a major architectural issue causing a double API call.

     Root Causes

     Bug 1: pollEvents() returns function reference instead of calling it 
     (CRITICAL)

     In cmd/nanobot/main.go:599-623, the pollEvents() method returns
     m.pollEvents (the method value) instead of m.pollEvents() (invoking it to
     get a tea.Cmd).

     In Bubble Tea, a tea.Cmd is func() tea.Msg. When pollEvents returns
     m.pollEvents (which is itself a func() tea.Cmd, not a tea.Msg), Bubble Tea
     treats it as a tea.Msg of type func() tea.Cmd — which no Update case
     matches. The polling loop silently stops, so no events ever reach
     handleEvent().

     Lines 605, 608, 613, 619 all have return m.pollEvents — should be return 
     m.pollEvents().

     Bug 2: Double API call wastes time and may cause inconsistency

     In internal/agent/loop.go:334-348, after streaming completes successfully,
     the code still calls Chat() again non-streaming:

     } else {
         // Still need to do a non-streaming call to get tool calls if any
         resp, chatErr = al.provider.Chat(ctx, messages, toolDefs, ...)

     Since StreamGenerate() is a fake-streaming wrapper around Chat(), this
     means every message triggers two identical API calls. The first call's
     content is streamed to the TUI, but the second call's resp is what gets
     used for tool calls and the final response. This doubles latency and could
     produce inconsistent results.

     Bug 3: pollEvents() architecture is fragile

     The current pollEvents() runs as a tea.Cmd (goroutine) that reads from
     channels and pushes to m.eventCh, then Update() checks m.eventCh via a
     non-blocking select. This is overly complex and has race conditions —
     events can be missed if Update() isn't called at the right time.

     A simpler approach: have pollEvents() directly return the event as a
     tea.Msg instead of routing through an intermediate channel.

     Plan

     Step 1: Fix pollEvents() to return events directly as tea.Msg

     File: cmd/nanobot/main.go

     Rewrite pollEvents() to directly return events as tea.Msg instead of
     routing through m.eventCh:

     func (m *interactiveModel) pollEvents() tea.Cmd {
         return func() tea.Msg {
             select {
             case ev := <-m.toolEventCh:
                 return toolEventMsg{ev: ev}
             case ev := <-m.agentEventCh:
                 return agentEventMsg{ev: ev}
             case resp, ok := <-m.outboundCh:
                 if !ok {
                     return nil
                 }
                 return responseMsg{content: resp.Content}
             case <-m.done:
                 return nil
             case <-time.After(50 * time.Millisecond):
                 return pollTickMsg{}
             }
         }
     }

     Add a pollTickMsg type and handle it in Update() to keep polling. Remove
     the eventCh field from interactiveModel and the intermediate channel logic.

     Update handleEvent() — it's no longer needed as a separate method. Events
     come directly through Update().

     Step 2: Fix the double API call in agent loop

     File: internal/agent/loop.go

     The StreamGenerate() fallback already calls Chat() internally. After
     streaming completes, the code should use the streamed content directly
     instead of calling Chat() again.

     Since StreamGenerate() doesn't return tool calls, and we need tool calls,
     the fix is:
     - Make StreamGenerate() return the full LLMResponse (including tool calls)
     on the final chunk, OR
     - Only call Chat() once and skip streaming entirely (simpler, since
     streaming is fake anyway)

     Simplest approach: since all providers' StreamGenerate() is just a wrapper
     around Chat(), skip the streaming path entirely and use Chat() directly,
     then publish the events manually:

     // In processMessage loop:
     al.publishAgentEvent(sessionKey, EventLLMThinking, nil)

     resp, err := al.provider.Chat(ctx, messages, toolDefs, opts)
     if err != nil { ... }

     // Publish events for TUI
     al.publishAgentEvent(sessionKey, EventLLMResponding, nil)
     al.publishAgentEvent(sessionKey, EventLLMStreamChunk, bus.StreamChunkData{
         Delta:    resp.Content,
         FullText: resp.Content,
     })
     al.publishAgentEvent(sessionKey, EventLLMFinal, bus.LLMFinalData{Content:
     resp.Content})

     This eliminates the double call, the 10-second timeout, and the complex
     streaming fallback logic.

     Step 3: Remove eventCh and simplify Update()

     File: cmd/nanobot/main.go

     - Remove eventCh field from interactiveModel
     - Remove the select block at the top of Update() that checks m.eventCh
     - Handle toolEventMsg, agentEventMsg, responseMsg, and pollTickMsg directly
      in the switch msg.(type) block
     - Each event handler should return m, m.pollEvents() to continue polling
     - Add pollTickMsg type that just returns m, m.pollEvents() to keep the loop
      alive

     Files to Modify

     1. cmd/nanobot/main.go — Fix pollEvents, simplify Update/handleEvent,
     remove eventCh
     2. internal/agent/loop.go — Remove double API call, simplify to single
     Chat() call

     appears instead of stuck on "Thinking..."
     3. Run single message mode: ./nanobot agent -m "Hello" — verify response
